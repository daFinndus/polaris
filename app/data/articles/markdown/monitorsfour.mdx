Back on track with valid admin credentials, the Cacti version was checked against known CVEs. Version <Code content="1.2.28" /> is vulnerable to <Marker content="CVE-2025-24367" link="https://github.com/SoftAndoWetto/CVE-2025-24367-PoC-Cacti" />, an authenticated RCE exploiting an arbitrary file write to drop a PHP webshell into the web root. The public PoC worked cleanly and produced a shell as <Code content="www-data" />. The user flag was readable directly from <Code content="/home/marcus/user.txt" /> due to misconfigured home directory permissions.

<br />

The environment was immediately unusual: no <Code content="netstat" />, no <Code content="ps" />, no <Code content="ssh" />. Manual checks confirmed something interesting: <Code content="uname" /> reported Linux, but the HTB machine was labelled Windows. Checking <Code content="/proc/mounts" /> revealed an overlay filesystem managed by <Code content="desktop-containerd" />, confirming the shell was inside a Linux container serving as the attack surface for the Windows host.

<br />

While reviewing the container's network configuration, <Code content="resolv.conf" /> revealed a nameserver at <Code content="192.168.65.7" />, which is the Docker host's internal gateway. The Docker daemon's default unauthenticated port was probed on a hunch:

<Block language="Bash" content={"curl 192.168.65.7:2375/version"} />

This returned a full Docker Engine version response. The daemon was completely unauthenticated and exposed to the container. This was the escape vector.

<br />

Available images were queried and <Code content="alpine:latest" /> was present on the host. A container was created via the API with the host root filesystem mounted at <Code content="/mnt/root" />, running <Code content="sleep infinity" /> to keep it alive:

The exec API was used to run commands inside the container. A <Code content="find" /> command located the root flag at:

A final exec call catted the file and captured the root flag.

<br />

In this machine I learned a lot and reinforced some things I already knew. Here are the main takeaways:

<br />

**Exhaust your current data before pivoting**: When the <Code content="/user" /> endpoint returned a full dump including real names, the right move was to derive usernames from that data — not jump straight to brute forcing. I wasted time on <Code content="Hydra" /> for something that was already solved.

<br />

**Check resolv.conf for internal network hints**: The Docker host's gateway IP was sitting in <Code content="resolv.conf" /> the whole time. Internal IPs in network config files can point directly to services that aren't externally exposed.

<br />

**Unauthenticated Docker API is an immediate escape**: If port <Code content="2375" /> is reachable, the game is over. No credentials needed — you can create containers, mount the host filesystem, and read anything. Always probe the Docker daemon when you find yourself in a container.

<br />

**printf over text editors**: In stripped environments with no <Code content="vim" />, <Code content="nano" />, or <Code content="echo" /> quirks, <Code content="printf" /> is the most reliable way to write files to disk. Kept me unblocked multiple times throughout this machine.

<br />

**LinPeas flags matter more than colors**: Yellow is noise. The percentage indicators — especially <Code content="99%" /> and <Code content="95%" /> — are what deserve immediate attention. Grep for those first, then go back to the red findings.
